# Phase 5.2 — Bulk Ingestion + Shareholding + Coverage — Design

**Date:** 2026-05-17
**Status:** Approved (awaiting written-spec review)
**Project:** ValueSight — Phase 5.2 (follows Phase 5.1 data layer foundation)
**Master plan:** `2026-05-17-master-architecture-plan.md` §4 (Group 4 shareholding), §5 (M1 ingestion bulk), §7 (Tier A)

## Problem

Phase 5.1 can ingest ONE ticker's financials on demand. To become a decision
engine across a ~1,500-stock universe (Tier A), we need to:

1. Capture **shareholding pattern** (promoter %, pledge %, FII, DII) — currently
   only available via the unreliable AI-extraction path
2. **Seed a universe** of tickers the user cares about
3. **Bulk-ingest** that universe efficiently, throttled to be polite to screener.in
4. Refresh it **nightly** without manual intervention
5. See **coverage** — which tickers are ingested and how fresh

- NSE official data (cookie/IP-block risk) — not pursued
- AI-extraction fallback removal → Phase 5.3
- Technical analysis, ranking, microtheories → later phases
- Real-time intraday shareholding → not meaningful (quarterly disclosure)

## Data Model

### New table: `company_shareholding`

screener.in's "Shareholding Pattern" section shows quarters as columns and
holder categories as rows. We store one row per (ticker, period_end).

```sql
CREATE TABLE IF NOT EXISTS company_shareholding (
  ticker          TEXT NOT NULL,
  period_end      DATE NOT NULL,
  period_label    TEXT,                -- e.g. 'Mar 2025'
  promoter_pct    NUMERIC,
  pledge_pct      NUMERIC,             -- % of promoter holding pledged
  fii_pct         NUMERIC,
  dii_pct         NUMERIC,
  government_pct  NUMERIC,
  public_pct      NUMERIC,
  shareholders    INTEGER,             -- number of shareholders, if shown
  source          TEXT DEFAULT 'screener.in',
  fetched_at      TIMESTAMPTZ DEFAULT NOW(),
  PRIMARY KEY (ticker, period_end)
);
CREATE INDEX IF NOT EXISTS idx_shareholding_ticker ON company_shareholding (ticker, period_end DESC);
```

### Extend `companies` with ingestion tracking

```sql
ALTER TABLE companies
  ADD COLUMN IF NOT EXISTS last_ingested_at TIMESTAMPTZ,
  ADD COLUMN IF NOT EXISTS ingest_status TEXT DEFAULT 'pending'
    CHECK (ingest_status IN ('pending','ok','failed')),
  ADD COLUMN IF NOT EXISTS ingest_error TEXT;
```

## Scraper Extension

`ingestion/screenerScraper.js` gains a `parseShareholding($)` step that reads
the `#shareholding` section. The screener.in shareholding table has rows like
"Promoters", "FIIs", "DIIs", "Government", "Public" and a "Pledged percentage"
sub-row. Output: one shareholding record per period column.

`parseScreenerHtml` return shape extends to include `shareholding: [...]`.

Pledge note: screener.in shows pledged % as a separate row under Promoters
(when present). The parser captures it into `pledge_pct`; null when absent.

## Bulk Ingestion (background processing)

New module `ingestion/bulkRunner.js`:

```
runBulkIngestion(tickers, db, opts) →
  - Processes tickers sequentially with a throttle (default 2,500ms between)
  - For each: calls ingestCompany() (existing) + ingestShareholding()
  - Updates companies.last_ingested_at, ingest_status, ingest_error per ticker
  - Returns when done; designed to run in background (fire-and-forget from HTTP)
```

Because ~50-500 tickers × 2.5s exceeds any HTTP timeout, the bulk endpoint
**starts the run in the background and returns immediately**. Progress is
observed via the coverage dashboard (reads `companies.last_ingested_at`).

A module-level `bulkState` object tracks `{ running, total, done, failed, startedAt }`
so the coverage dashboard can show live progress. If the Node process restarts
mid-run (e.g. Render redeploy), the run dies gracefully — already-ingested
tickers are persisted; re-running picks up where it left off (stalest first).

## Universe Seed

Admin pastes a newline/comma-separated ticker list. Endpoint inserts each into
`companies` with `ingest_status='pending'`. Existing tickers are left intact
(upsert on ticker, no overwrite of already-ingested data).

```
POST /api/admin/universe/seed  { tickers: "HFCL, BHARTIARTL, TCS, ..." }
  → { added: N, alreadyPresent: M }
```

## Nightly Cron

Extend the existing cron orchestration. New endpoint:

```
POST /api/cron/ingest-universe   (header x-cron-secret)
  → selects the N stalest active tickers (oldest last_ingested_at, nulls first)
  → starts a background bulk run for them
  → returns { started: true, batch: N } immediately
```

Batch size `N` defaults to 50 (configurable via `INGEST_BATCH_SIZE` env). At
50/night a 500-ticker universe fully refreshes every ~10 days, which is fine
for fundamentals that change quarterly.

cron-job.org pings this nightly (separate schedule from the existing price-check).

## API Surface

| Method + Path | Auth | Purpose |
| --- | --- | --- |
| `POST /api/admin/universe/seed` | admin | Seed companies from pasted ticker list |
| `POST /api/admin/ingest/bulk` | admin | Start background bulk run for given tickers or all stale |
| `GET /api/admin/ingest/status` | admin | Live `bulkState` + coverage summary |
| `POST /api/cron/ingest-universe` | cron secret | Nightly batch of stalest tickers |

## Acceptance Criteria

1. Pasting a Nifty 500 list seeds `companies` with `ingest_status='pending'`.
2. Clicking "Ingest all stale" starts a background run; coverage dashboard
   shows live progress and final per-ticker status.
3. Each successfully ingested ticker has annual P&L/BS/CF, quarterly P&L,
   derived rows, aggregates, AND a `company_shareholding` row.
4. Promoter pledge for a pledged company (e.g. a stock with known pledge)
   appears correctly in `company_shareholding.pledge_pct`.
5. Nightly cron processes the 50 stalest tickers without HTTP timeout.
6. AI-extraction fallback still works for tickers not yet ingested.
7. `test/shareholdingParser.test.js` and `test/bulkRunner.test.js` pass.

## Risk Notes

- **screener.in rate limiting:** 2.5s throttle + realistic User-Agent. If they
  block, the run records failures per ticker and continues; retried next night.
- **Background job dies on redeploy:** acceptable — persisted progress means a
  re-run resumes from stalest. No data corruption (idempotent upserts).
- **Shareholding HTML variance:** some companies show pledge, some don't; some
  show government holding, some don't. Parser treats every field as nullable.
